# Aeroric 代码优化与组件整合提示词

本文档用于指导后续智能体扫描 Aeroric 项目前后端，在不影响现有功能的前提下做代码优化、组件整合和可维护性改进。

## 项目背景

- 项目名称：Aeroric
- 项目定位：面向 AI 编程智能体的桌面工作台
- 前端技术栈：React 19、TypeScript、Vite、Vitest、ESLint、Prettier、Tauri API
- 后端技术栈：Tauri 2、Rust、Tokio、rusqlite、dbx-core
- 主要前端目录：
  - `src/components/`：页面与功能组件
  - `src/hooks/`：状态管理和副作用逻辑
  - `src/lib/`：跨组件业务 API、数据转换、协议逻辑
  - `src/styles/`：样式模块
  - `src/types/`：共享类型
  - `src/test/`：Vitest 测试
- 主要后端目录：
  - `src-tauri/src/`：Tauri 命令、系统能力、持久化、PTY、SSH、Git、数据库能力
  - `src-tauri/src/database/`：数据库连接、查询、结构、Redis、Mongo、导入导出等能力
- 需要重点关注的高风险区域：
  - `src/components/database/DatabaseView.tsx`：体量很大，优先做边界梳理和渐进拆分
  - `src/components/database/RedisBrowser.tsx`、`src/components/database/MongoBrowser.tsx`：浏览器交互逻辑和状态逻辑可继续收敛
  - `src/lib/databaseApi.ts` 与 `src/types/database.ts`：前后端数据契约应保持集中、明确、可测试
  - `src-tauri/src/database/*.rs`：后端数据库命令需要保持错误处理、类型转换和命令边界一致

## 完整提示词

将下面提示词复制给负责执行优化的智能体：

```text
你是一个资深全栈代码维护工程师。请扫描当前 Aeroric 项目的前端和后端代码，在不影响现有功能、不改变用户可见行为、不覆盖未提交改动的前提下，做代码优化和组件整合，目标是提升可维护性、可读性、模块边界清晰度和测试可信度。

项目上下文：
- 工作目录是 Aeroric 项目根目录。
- 这是一个 Tauri 2 + Rust 后端、React 19 + TypeScript + Vite 前端的桌面应用。
- 前端核心目录包括 `src/components/`、`src/hooks/`、`src/lib/`、`src/styles/`、`src/types/`、`src/test/`。
- 后端核心目录包括 `src-tauri/src/` 和 `src-tauri/src/database/`。
- 当前工作区可能存在未提交改动。开始前必须运行 `git status --short`，明确哪些文件已经被修改。不要回滚、覆盖或格式化无关文件。只修改本次优化确实需要修改的文件。
- 如果目标文件已有未提交改动，必须先记录这些改动的范围，并在后续编辑中只做最小增量修改，避免覆盖用户或其他智能体已经完成的工作。

总体目标：
1. 在行为不变的前提下优化代码结构。
2. 合并重复逻辑，抽出清晰的工具函数、hooks 或小组件。
3. 拆分过大的组件，但保持对外 props、事件流和 UI 行为稳定。
4. 统一类型定义和前后端数据契约，减少 `any`、重复 interface、重复字符串常量。
5. 保持错误处理、空状态、加载状态、取消/清理逻辑完整。
6. 补充或调整必要测试，确保重构有测试保护。
7. 避免无关重构、风格大改、依赖升级、设计改版和大规模格式化。

执行步骤：

第一步：基线扫描
- 运行：
  - `git status --short`
  - `git diff --name-only`
  - `git diff --stat`
  - `pnpm lint`
  - `pnpm test`
  - `pnpm build`
- 如果 `git diff --name-only` 显示目标文件已经存在未提交改动，只允许在明确必要时修改这些文件；修改前必须在小范围计划中说明原因、影响范围和如何避免覆盖既有改动。
- 如果基线测试或构建已经失败，先记录失败项和错误摘要，不要把既有失败归因于本次优化。
- 使用 `rg`、`find`、`wc -l` 等命令识别高复杂度文件、重复逻辑和跨模块耦合点。
- 优先关注：
  - `src/components/database/DatabaseView.tsx`
  - `src/components/database/RedisBrowser.tsx`
  - `src/components/database/MongoBrowser.tsx`
  - `src/hooks/useRedisBrowser.ts`
  - `src/hooks/useMongoBrowser.ts`
  - `src/lib/databaseApi.ts`
  - `src/types/database.ts`
  - `src-tauri/src/database/*.rs`

第二步：制定小范围优化计划
- 不要一次性重构整个项目。
- 每一轮只选择 1 个功能域或 1 个稳定边界，优先保证改动可验证、可回退。
- 不要在同一轮同时修改前端展示层、前端 API 契约和 Rust 后端 command；跨层改动必须拆成多轮推进。
- 每个优化点必须说明：
  - 当前问题是什么
  - 预期收益是什么
  - 会影响哪些文件
  - 目标文件是否已有未提交改动
  - 如何验证行为不变
- 优先选择这些类型的优化：
  - 从大组件中抽出纯展示组件
  - 从组件中抽出稳定的状态派生逻辑
  - 合并重复按钮、列表项、空状态、错误状态、搜索面板或树节点逻辑
  - 将散落的数据库命令名、类型转换、字段映射集中到 `src/lib/` 或 `src/types/`
  - 将 Rust 后端重复的错误映射、连接查找、参数校验封装为小函数

第三步：前端优化规则
- 组件拆分必须以“稳定边界”为准，而不是为了拆分而拆分。
- 新组件应满足以下条件之一：
  - 有独立展示职责
  - 有可复用交互模式
  - 能显著降低父组件认知负担
  - 能被单独测试
- hooks 应只承载状态、派生数据、副作用和回调编排，不应混入大量 JSX。
- `src/lib/` 应承载与 UI 无关的协议、转换、解析、命令调用封装。
- `src/types/` 应承载跨模块共享类型，局部类型优先留在局部文件。
- 保留现有 UI 样式和交互，不做视觉重设计。
- 不新增不必要的依赖。
- 不把多个小而清晰的文件强行合并成大文件。
- 不引入过度抽象，例如泛型工厂、复杂配置驱动 UI、难以追踪的高阶组件。

第四步：后端优化规则
- Tauri command 的入参、返回值和错误字符串要保持向后兼容，除非同步更新前端和测试。
- Rust 模块拆分应围绕数据库连接、查询、结构、导入导出、Redis、Mongo 等领域边界。
- 将重复的连接查找、错误转换、参数校验抽成私有 helper。
- 不改变数据库持久化格式、配置路径、事件名称、命令名称和前端依赖的 JSON 字段。
- 不做依赖升级和大规模架构迁移。
- 任何后端行为调整必须有前端调用侧或 Rust/集成测试保护。

第五步：测试策略
- 对纯函数、数据转换、命令参数构造、树结构转换、搜索过滤、排序分页等逻辑优先补单元测试。
- 对大组件拆分，优先补充或确认 characterization tests，先锁定用户可见行为，再移动代码。
- 对组件拆分，优先复用现有测试断言用户可见行为，而不是测试内部实现。
- 修改数据库浏览、Redis、Mongo、文件运行、终端输入等功能时，必须运行相关测试文件。
- 如果新增测试成本低，应补充测试覆盖重构后的关键边界。
- 禁止删除测试来让构建通过。

数据库相关重构的最小回归清单：
- 数据库连接列表、选中连接和连接状态展示保持稳定。
- SQL 查询、表结构查看、结果表格渲染、分页或排序行为保持稳定。
- Redis key 搜索、树节点展开、值查看、刷新、错误状态和空状态保持稳定。
- Mongo collection 列表、document 查看、搜索或刷新、错误状态和空状态保持稳定。
- 加载状态、取消或清理逻辑不丢失，不引入重复请求或 stale state。

第六步：验证命令
- 修改完成后至少运行：
  - `pnpm lint`
  - `pnpm test`
  - `pnpm build`
- 如果改动涉及 Rust 后端，额外运行：
  - `cargo check --manifest-path src-tauri/Cargo.toml`
- 如果只改文档，可不运行完整构建，但要说明未运行原因。
- 最终报告必须列出实际运行过的命令和结果。

第七步：输出要求
- 最终回复使用中文简体。
- 先说明完成了哪些优化，再说明验证结果。
- 如果有未解决问题，明确列出文件、原因和建议下一步。
- 不要声称“全部通过”，除非相关命令确实运行并通过。
- 不要覆盖用户未提交改动，不要执行 `git reset --hard`、`git checkout --` 等破坏性命令。

推荐优先优化顺序：
1. 数据库前端视图：先围绕 `DatabaseView.tsx` 拆分展示层、操作栏、状态面板、结果区域，不改变现有 hooks 和 API。
2. 数据库浏览器逻辑：对 Redis/Mongo Browser 的搜索、树结构、加载状态、错误状态做共享工具或局部组件整合。
3. 前后端数据库契约：收敛 `databaseApi.ts` 和 `types/database.ts` 中重复类型、命令调用和字段映射。
4. Rust 数据库模块：提取重复错误处理和连接查找 helper，保持 command 签名稳定。
5. 通用 UI 小组件：整合项目中重复的 icon button、空状态、列表行、搜索输入、状态标签，但不要破坏已有样式。

请从“基线扫描”开始执行，并在每一轮只做小步、可验证、可回退的优化。
```

## 操作步骤清单

后续真正执行代码优化时，建议按下面顺序推进：

1. 保护现场  
   运行 `git status --short`、`git diff --name-only`、`git diff --stat`，确认当前未提交改动。不要覆盖他人或用户已有改动。目标文件已有改动时，只做明确必要的最小增量修改。

2. 建立基线  
   运行 `pnpm lint`、`pnpm test`、`pnpm build`。如涉及后端，运行 `cargo check --manifest-path src-tauri/Cargo.toml`。

3. 选择小范围目标  
   不要一次性重构所有前后端。每轮只选择一个大文件中的稳定边界或一个功能域，例如数据库视图展示层、Redis 浏览器搜索逻辑、Mongo 浏览器状态派生逻辑或数据库 API 类型。

4. 先补测试或确认现有测试  
   对准备重构的行为找到对应测试。没有测试时，优先为高风险纯逻辑或用户可见行为补 characterization tests。

5. 渐进重构  
   每次只移动一类逻辑：展示组件、状态派生、API 封装、类型定义或后端 helper。移动后立即运行相关测试。

6. 控制改动范围  
   不做依赖升级，不做视觉重设计，不做全仓格式化，不修改无关文件。

7. 完整验证  
   完成后运行 lint、test、build。涉及 Rust 后端时运行 cargo check。

8. 输出总结  
   说明改动文件、优化收益、验证命令和剩余风险。

## 优化验收标准

- 用户可见行为保持不变。
- 未提交的既有改动没有被回滚或覆盖。
- 大组件职责更清晰，父组件代码量和分支复杂度下降。
- 重复逻辑被合并到合适层级，没有形成过度抽象。
- 类型定义更集中，跨前后端字段映射更明确。
- 错误处理、加载状态、空状态和清理逻辑没有丢失。
- 相关测试通过，或明确记录基线已有失败。

## 禁止事项

- 禁止在没有用户明确要求时执行破坏性 Git 命令。
- 禁止为了重构而改变功能、交互、数据格式、命令名称或事件名称。
- 禁止大规模格式化无关文件。
- 禁止删除测试、降低 lint 标准或绕过类型检查。
- 禁止引入与当前技术栈不一致的新框架。
- 禁止把清晰的小模块合并成难以维护的大模块。
